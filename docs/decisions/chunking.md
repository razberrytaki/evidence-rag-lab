# 결정: 청킹

## 맥락

작은 청크는 검색 정밀도를 높이지만, 큰 청크는 문맥 보존에 유리하다. 청크 경계가
흐릿하면 인용도 흐릿해진다. 같은 문장이 다른 청크 id로 이동하면 이전 추적 기록과
평가 사례가 설명력을 잃는다.

현재 샘플 실행은 문서가 작기 때문에 원본 문서당 청크 1개를 만든다. 작은 공개 문서 세트에서는
검색, 인용, 신뢰도, 추적 기록 경계를 먼저 검증하고, 청킹 전략은 확장 단계에서
독립 실험으로 키운다.

## 권장 선택

MVP 문서에는 원본 문서 단위 청크를 사용한다. 다음 단계의 기본 후보는 제목
구조를 인식하는 재귀적 청킹이다. 목표는 고정 크기 분할보다 제목 경로와 안정적인
청크 id를 더 잘 보존하는 것이다.

## 검토한 대안

- 고정 크기 청킹
- 시맨틱 청킹
- 부모-자식 검색

## 트레이드오프

원본 문서 단위 청크는 단순하고 인용을 검사하기 쉽다. 대신 긴 문서에서는
잡음이 늘고, 한 문서 안의 서로 다른 주장들이 같은 검색 단위에 묶인다.

고정 크기 청킹은 구현이 쉽지만 제목, 코드 블록, 표 경계를 끊을 수 있다. 그
경우 인용 문구가 같은 주장을 안정적으로 가리키기 어렵다.

제목 기반 재귀 청킹은 사람이 검사하기 충분히 단순하고, 공개 기술 문서의 제목
구조와 잘 맞는다. 부모 문맥을 함께 두면 자식 청크는 검색 정밀도를 맡고,
부모 문맥은 생성 근거를 보강할 수 있다.

시맨틱 청킹은 문장 의미를 더 잘 보존할 가능성이 있지만 모델 의존성, 재현성 저하,
경계 디버깅 비용을 추가한다. 이 프로젝트의 우선순위는 먼저 결정적 검색/평가/추적
경계를 세우는 것이다.

## 평가 근거

- `packages/ingest/src/sample-docs.ts`는 현재 샘플 문서를 원본 문서당 청크
  1개로 로드한다.
- `packages/ingest/test/sample-docs.test.ts`는 20개 샘플 문서와 `#chunk-001`
  계약을 검증한다.
- 검색 품질 사례에는 `chunking-boundary`와 `parent-child-retrieval`이 있다.
  현재 검증은 청킹 알고리즘 자체가 아니라, 청킹/부모-자식 검색 결정을
  설명하는 문서가 검색되는지에 가깝다.
- `DocumentChunk`는 `headingPath`, `contentHash`, `version`, optional `parentId`를
  갖는다. 이후 재귀적 청킹과 부모-자식 검색으로 확장할 수 있는 스키마
  자리는 마련되어 있다.

## 확장 시 다시 볼 것

대규모로 확장할 때는 문서당 평균 청크 수가 바로 색인 크기와 추적 기록량으로 번진다.
현재 확장성 예산은 문서 `10,000,000`개와 문서당 평균 `8`개 청크를 가정해
`80,000,000`개 청크를 계산한다. 이후에는 겹침 폭, 제목 깊이, 부모 문맥 크기,
청크 id 이동, 색인 보정 비용, 인용 안정성을 함께 측정한다.
